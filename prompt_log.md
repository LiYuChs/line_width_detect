# 專案紀錄：微米級線寬量測演算法開發 (Q&A)

## 一、 基礎量測與傳統演算法建置

**Q1: 影像中有 10 條在黑色色板上的線，如何用程式精準計算並標記每一條線的寬度？**
**A1:** 由於線條寬度極細（約 3~4 像素）且邊緣模糊，直接使用肉眼或二值化（Thresholding）計算像素點會產生極大誤差。
解決方案為導入「一維訊號處理」：
1. 擷取畫面正中央垂直切線的灰階亮度分佈（Intensity Profile）。
2. 使用 `scipy.signal.find_peaks` 演算法穩定抓取 10 條線的發光中心。
3. 利用半高寬（FWHM）數學模型，計算出具備小數點精度的「次像素（Sub-pixel）」寬度。

**Q2: 如何將單純的像素寬度換算為實際的物理距離（如 mm）？**
**A2:** 需要已知標準件作為基準。利用影像中前四條已標註實際寬度（0.015mm, 0.016mm）的線段，將其「實際物理寬度」除以演算法測得的「次像素寬度」。對這些數據取平均值後，即可得出一組穩定的環境比例尺（例如此專案推算約為 0.00363 mm/pixel）。將此比例尺乘上所有測得的像素寬度，即可得到實際物理寬度。

**Q3: 為了未來的程式擴充性（如批次處理）與產出報告，程式架構該如何設計？**
**A3:** 應採用物件導向（OOP）進行模組化解耦：
* `ImageAnalyzer`: 專職影像處理、動態尋找 ROI（黑色色板區域）與 FWHM 像素寬度計算。
* `Calibrator`: 專職比例尺計算。
* `DataFormatter`: 專職視覺化疊圖（Overlay）與匯出 Pandas DataFrame 至 Excel 總表。
* `BatchProcessor`: 負責遍歷資料夾、處理例外狀況，並將單張影像的結果合併為 Master Excel Report。

## 二、 深度學習架構導入與優化

**Q4: 若想使用 SAM (Segment Anything Model) 進行線條分割，會有什麼限制？**
**A4:** SAM 是強大的 Zero-shot 語意分割模型，但在微米級工業影像上有兩個主要限制：
1. **精度降級**：SAM 輸出的是二值化遮罩（Binary Mask），量測結果會退化為整數像素，流失 FWHM 的次像素精度。
2. **語意沾黏**：面對極細、模糊且密集的平行線，SAM 傾向將其判定為單一具備條紋的色塊，導致 10 條線沾黏在一起，需要精準的提示工程（Prompt Engineering）才能分開。

**Q5: 希望使用深度學習推論（Inference）且不重新訓練模型，除了 SAM 還有什麼好方法？**
**A5:** 結合「AI 超解析度 (Super Resolution)」與「SAM」是最佳解。
具體流程為：手動或自動裁切感興趣區域 (ROI) -> 使用 Real-ESRGAN 將影像無損放大 4 倍並銳化 -> 將高解析度影像餵給 SAM 進行分割。
此架構的強大之處在於：SAM 在放大 4 倍的影像上運算，當寬度換算回原圖比例時，原本 SAM 1 像素的誤差會縮小為 0.25 像素，變相幫 SAM 找回了流失的「次像素精度」。

**Q6: 即使使用了 AI 超解析度放大影像，還需要依賴傳統的 `find_peaks` 演算法嗎？**
**A6:** 絕對需要。SAM 的「全自動模式 (Everything Mode)」容易落入語意陷阱，將 10 條清晰的平行線群聚辨識為「一個斑馬紋矩形」。`find_peaks` 扮演了無可取代的「自動提示器 (Auto-Prompter)」，它能在毫秒間提供 10 個極度精準的中心座標，引導 SAM 精確分割指定物件。這展示了「傳統演算法做定位，深度學習做特徵提取」的完美互補。

**Q7: 如何解決 SAM 在密集線條中依然發生的「沾黏」問題？**
**A7:** 需要結合「形態學」與「強約束提示工程」：
1. **形態學閉合 (Morphological Closing)**：在 SR 放大後，使用 3x3 核心進行輕微的閉合運算，填補線條的細微斷點，讓物件語意更連續。
2. **Y 軸負向提示點 (Negative Prompts)**：在每條線的「上方」與「下方」加入不要選取（Label=0）的點，強迫 SAM 區分線條與間距。
3. **邊界框約束 (Bounding Box)**：根據放大後的物理線距（約 40px），程式化生成高度為 30px 的扁平邊界框。這能在幾何空間上徹底阻絕 SAM 跨越間距去沾黏相鄰的線條。

## 三、 環境建置與除錯

**Q8: 在 Windows 安裝 Real-ESRGAN 的相依套件 `lmdb` 時發生編譯失敗，如何解決？**
**A8:** 1. **缺乏編譯器**：需下載並安裝 Microsoft Visual C++ Build Tools，勾選「使用 C++ 的桌面開發」與「Windows SDK」。
2. **路徑長度限制**：Windows 有 MAX_PATH (260字元) 限制。需在命令列建立並指向短路徑暫存資料夾（如 `set TMP=C:\tmp`）以繞過此問題。
3. **版本相容性**：最新版 `lmdb` 可能不支援舊版 Python（如 Python 3.8）的 C API。可指定降級安裝相容版本（如 `pip install lmdb==1.4.1`）。
